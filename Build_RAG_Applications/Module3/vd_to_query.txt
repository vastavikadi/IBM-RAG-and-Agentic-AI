- ​Let's look at how you can generate embeddings. ​To generate and store embeddings, LlamaIndex uses the VectorStoreIndex class. ​For simple use cases, the embeddings can be generated using a default model and stored ​in-memory by simply passing in the nodes as the only parameter to VectorStoreIndex. ​For more complex cases that require a custom embedding model and persistent storage, ​the following procedure is needed. First, import the necessary libraries. ​In this case, you are importing the ChromaDB vector database to store the embeddings ​and a function that imports embedding models from HuggingFace.

- ​Then define the embedding model. In this case, you are using an embedding model from HuggingFace. ​Next, set up your vector store and storage context. In this case, you are using ChromaDB and then ​passing the embedding model and storage context into VectorStoreIndex. ​Then pass the nodes, the embedding model, and the storage context ​into VectorStoreIndex to generate embeddings using the selected model ​and to store the embedding vectors inside the persistent storage defined by the storage context. ​Thus, whether you store embedding vectors in-memory or in persistent storage, ​LlamaIndex's powerful VectorStoreIndex class efficiently handles both embedding text chunks ​and storing the resulting embeddings in the vector store. ​Given a VectorStoreIndex, the user's original prompt, embedding, ​and retrieval steps can be combined in LlamaIndex using the following process.

- ​First, create the retriever by calling the "as_retriever" method on the VectorStoreIndex ​object. Then pass the user's prompt to the retriever object's retrieve method to obtain ​the relevant results. Alternatively, if you need to control the maximum number of items retrieved, ​specify the maximum number of retrieved items by setting the similarity_top_k ​parameter when creating the retriever object. ​When the user's prompt is passed to this retriever, it retrieves the k nodes most ​similar to the user's prompt from the vector store, which in this case would be 5 nodes. The retrieved ​nodes are returned as a ranked list with the most similar nodes at the top of the list. 

- ​Let's now discuss prompt augmentation, LLM querying, and response generation. ​LlamaIndex has the ability to combine the prompt augmentation, ​LLM querying, and response generation RAG steps using a response synthesizer. 
​Given a user's prompt and retrieved nodes, the response synthesizer's synthesize method ​generates a response from the LLM. Note that the embedding of the user's prompt, ​prompt augmentation, and the passing of the augmented prompt to the LLM are done in the ​background, and no additional steps have to be taken by the user to perform these steps manually. ​With LlamaIndex, the process can be simplified even further using a query engine. ​LlamaIndex's query engine combines the prompt embedding, ​retrieval, prompt augmentation, LLM querying, and response generation steps, ​eliminating the need to manually retrieve nodes and to pass them to a response synthesizer. ​Given a user's prompt, the query engine's query method generates a response from the LLM. ​Note that the embeddings of the user's prompt, retrieval, prompt augmentation, ​and the passing of the augmented prompt to the LLM are done in the background, ​and no additional steps have to be taken by the user to perform these tasks manually. 

- ​As a result, LlamaIndex's query engines greatly reduce the amount of code that needs to be ​written when building a RAG application.
